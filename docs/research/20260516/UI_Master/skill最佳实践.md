# Skill 最佳实践

## 一、SKILL.md 标准格式

### 1.1 文件结构

```
.trae/skills/<skill-name>/
└── SKILL.md
```

### 1.2 SKILL.md 格式规范

```yaml
---
name: "<skill-name>"
description: "<核心定位>。专注于<具体领域>——<能力概述>。紧密结合<领域绑定>。触发条件：当需要<何时调用>时调用。"
---
```

```markdown
# [Skill名称] — [角色定位]

## 身份定义
你是一位[角色定位]。你的核心能力是[核心能力描述]。

## 核心原则
### 原则1：[原则名]
[原则描述]

## 领域知识
（领域专属知识，让Agent具备专家级理解）

## 架构设计
（核心输出结构）

## Anti-Patterns
| 禁令 | 理由 |
|------|------|
| ... | ... |

## 工作模式
### Mode 1：Quick
### Mode 2：Production

## 执行步骤
1. **步骤1**：...
2. **步骤2**：...

## 默认行为
如果调用时没有额外context，先询问：...
不假设，先询问。
```

## 二、Frontmatter 字段规范

### 2.1 name 字段

| 规则 | 说明 |
|------|------|
| 格式 | kebab-case（用`-`连接），如 `gcs-graph-visualizer` |
| 唯一性 | 全局唯一标识符，不可重复 |
| 简洁性 | 体现核心定位，不宜过长 |
| 命名模式 | `[项目前缀]-[领域]-[角色]` |

### 2.2 description 字段

| 规则 | 说明 |
|------|------|
| 长度 | ≤200字符（超出影响显示和匹配） |
| 四段式 | 定位+能力+绑定+触发条件 |
| 触发条件 | 必须用"触发条件：当需要...时调用"固定句式 |
| 格式 | YAML多行文本 `|` 或单行字符串 |
| 可匹配 | Agent根据description决定是否调用，必须具体可匹配 |

**四段式模板**：
```
<核心定位>。专注于<具体领域>——<能力概述>。紧密结合<领域绑定>。触发条件：当需要<何时调用>时调用。
```

**优秀案例**：
```
设计系统大师级UI架构师。在编写任何UI代码之前，先构建完整的设计系统作为语义契约。适用于从0到1构建产品界面、建立团队设计规范。触发条件：当需要创建新项目UI、建立设计系统时调用。
```

## 三、正文各节段规范

| 节段 | 必须 | 设计原理 | 最佳实践 |
|------|------|----------|----------|
| **身份定义** | ✓ | 给Agent角色锚点，约束行为边界 | 一句话说清"你是谁"+"核心能力" |
| **核心原则** | ✓ | 3-5条不可妥协的原则，决策基准 | 每条原则有名称+描述+具体要求 |
| **领域知识** | 推荐 | 让Agent具备专家知识 | 只写与skill直接相关的领域知识 |
| **架构设计** | ✓ | 核心输出结构 | 用代码块/表格/图示，不用散文 |
| **Anti-Patterns** | ✓ | 禁令比指令更有效 | 表格形式：禁令+理由 |
| **工作模式** | 推荐 | 不同场景不同严格度 | 至少Quick/Production两种 |
| **执行步骤** | ✓ | 标准化流程，可重复可验证 | 有序、每步有明确产出 |
| **默认行为** | ✓ | 防止Agent自作主张 | "不假设，先询问"原则 |

## 四、核心设计原则

### 4.1 禁令优于指令

明确"不做"什么比"做"什么更能约束AI行为。Anti-Patterns表格是skill的核心防护机制。

### 4.2 询问优于假设

没有足够context时不假设，而是询问用户。默认行为必须是"先询问"。

### 4.3 理由注释

每个设计决策附理由注释，让Agent理解"为什么"而非仅知道"是什么"。

### 4.4 渐进式加载

先全局后局部，避免token浪费。skill正文应包含最关键信息，细节按需加载。

### 4.5 Non-Negotiables

不可协商的约束是skill的核心保障。这些约束在任何工作模式下都不可违反。

### 4.6 可组合性

Skill应该可以与其他skill协同工作。通过5号调度中枢实现组合，1-4号不直接互相调用。

## 五、Skill的技术本质

```
Skill = 规则(Rules) + 上下文(Context) + 工具(Tools)
```

- **Rules**：解决"怎么做"（标准与约束）
- **Context**：解决"在哪做"（代码、文档、当前状态）
- **Tools**：解决"拿什么做"（外部连接与能力）

## 六、评估与迭代

### 6.1 三代理评估

1. **Grader**：按评分标准逐项打分
2. **Comparator**：有skill vs 无skill盲评比较
3. **Analyzer**：多轮评分后汇总分析规律

### 6.2 防过拟合

- Train/Test分割（60/40）
- 训练集上迭代改进
- 测试集验证泛化能力

### 6.3 描述触发优化

- 自动生成多个版本的description
- 用测试句子逐版本测触发准确率
- 选择触发准确率最高的版本

## 七、格式对比（Trae vs 其他工具）

| 维度 | Trae Skills | Claude Code | Cursor Rules | GitHub Copilot |
|------|-------------|-------------|--------------|----------------|
| 元数据 | name + description | 无 | 无 | 无 |
| 触发机制 | description匹配 | 始终激活 | 始终激活 | 始终激活 |
| 可组合性 | 高 | 中 | 低 | 低 |
| 工作模式 | 支持多模式 | 不支持 | 不支持 | 不支持 |
| Anti-Patterns | 内置机制 | 无 | 无 | 无 |

Trae Skills通过frontmatter的description实现了**按需触发**，通过name实现了**可组合调用**，这是其他工具不具备的。
