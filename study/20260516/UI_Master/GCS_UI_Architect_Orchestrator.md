# GCS UI架构师调度协议

## 一、问题定义

5个V2架构师各自覆盖GCS管线的不同阶段，存在重叠区域。核心问题：

```
GCS Pipeline:  IO → Core → DCM → LGS → [Prop] → CDS → App
                    │      │      │      │       │      │
1号(图可视化):      │      ●──────│──────│───────│      │
2号(管线仪表):      │      │      │      ●───────●──────●
3号(几何空间):      ●──────│──────│──────│───────●      │
4号(图工程):   ●────│──────│──────│──────│───────│      │
5号(全栈平台): ●────●──────●──────●──────●───────●──────●
```

**重叠矩阵**：

| 管线阶段 | 1号 | 2号 | 3号 | 4号 | 5号 | 重叠度 |
|----------|-----|-----|-----|-----|-----|--------|
| IO       |     |     |     | ●   | ●   | 2 |
| Core     |     |     | ●   |     | ●   | 2 |
| DCM      | ●   |     |     |     | ●   | 2 |
| LGS      | ●   | ●   |     |     | ●   | 3 |
| Prop     |     | ●   |     |     | ●   | 2 |
| CDS      |     | ●   | ●   |     | ●   | 3 |
| App      |     | ●   |     |     | ●   | 2 |

**关键重叠区**：
- **LGS**：1号(图论视角DOF) vs 2号(数值视角DOF) vs 5号(代数规范DOF)
- **CDS**：2号(收敛分析) vs 3号(求解动画) vs 5号(管线编排)

## 二、调度模型：三层架构

### 层1：5号作为调度中枢（Orchestrator）

**核心原则**：5号（系统架构师）是唯一的调度中枢，其他4个架构师不直接互相调用。

```
                    ┌──────────────┐
                    │   5号        │
                    │  系统架构师   │
                    │  (Orchestrator)│
                    └──────┬───────┘
                           │
              ┌────────────┼────────────┐
              │            │            │
        ┌─────┴─────┐ ┌───┴───┐ ┌─────┴─────┐
        │   1号     │ │  2号  │ │   3号     │
        │ 图可视化  │ │管线仪表│ │ 几何空间  │
        └───────────┘ └───────┘ └───────────┘
              │
        ┌─────┴─────┐
        │   4号     │
        │ 图工程    │
        └───────────┘
```

**理由**：
- 5号覆盖所有管线阶段，具备全局视野
- 5号基于代数规范和信息代数，天然适合做信息路由
- 避免环形依赖：如果1号直接调用2号，2号又调用3号，3号又调用1号，形成环

### 层2：1-4号作为领域专家（Domain Experts）

1-4号架构师是**领域专家**，各自拥有独立的理论基础和专业视角：
- 1号：代数图论专家——"图的拓扑结构说什么？"
- 2号：数值分析专家——"求解器在做什么？"
- 3号：画法几何专家——"几何空间长什么样？"
- 4号：图论构造专家——"如何构造/修复图？"

### 层3：共享数据契约（Shared Data Contract）

所有架构师通过**共享数据契约**通信，而非直接调用：

```
┌─────────────────────────────────────────┐
│        Shared Data Contract             │
│                                         │
│  ┌───────────┐  ┌───────────┐          │
│  │ Manager   │  │ Decomposi-│          │
│  │ (Core)    │  │ tionResult│          │
│  └───────────┘  └───────────┘          │
│  ┌───────────┐  ┌───────────┐          │
│  │ StatusRe- │  │ SolverRe- │          │
│  │ port      │  │ port      │          │
│  └───────────┘  └───────────┘          │
│  ┌───────────┐  ┌───────────┐          │
│  │ UIState   │  │ ViewCon-  │          │
│  │           │  │ fig       │          │
│  └───────────┘  └───────────┘          │
└─────────────────────────────────────────┘
```

## 三、调度协议

### 3.1 用户意图→架构师调度矩阵

| 用户意图 | 主架构师 | 辅助架构师 | 调度模式 |
|----------|----------|-----------|----------|
| "我想看约束图的结构" | 1号 | — | 独立 |
| "我想分析图的刚性" | 1号 | 4号(如需修复) | 顺序 |
| "我想监控求解过程" | 2号 | — | 独立 |
| "我想看几何体的3D位置" | 3号 | — | 独立 |
| "我想生成一个约束图" | 4号 | 1号(验证可视化) | 顺序 |
| "我想修复一个有问题的图" | 4号 | 1号(验证) + 2号(DOF分析) | 并行→汇聚 |
| "我想构建完整的GCS平台" | 5号 | 1+2+3+4号 | 分层组合 |
| "我想从输入到求解走一遍" | 5号 | 4→1→2→3 | 管线顺序 |
| "求解失败了，帮我诊断" | 2号 | 1号(图结构) + 3号(几何位置) | 并行→汇聚 |
| "我想对比两个场景" | 5号 | 1+2+3号(各维度对比) | 并行→汇聚 |

### 3.2 调度模式定义

#### 模式A：独立模式（Independent）

```
用户意图 → [单一架构师] → 输出
```

适用场景：用户需求明确落在单一架构师的专业领域内。

规则：
- 主架构师独立完成所有决策
- 不需要与其他架构师协调
- 输出完全由主架构师的理论基础决定

#### 模式B：顺序模式（Sequential）

```
用户意图 → [架构师A] → 中间产物 → [架构师B] → 输出
```

适用场景：前一个架构师的输出是后一个架构师的输入。

规则：
- A完成后，5号将A的输出包装为B的输入
- B不知道A的存在，只看到标准数据契约
- A→B的交接由5号负责数据格式转换

交接协议：
```
5号收到A的输出 → 验证数据契约合规 → 注入B的上下文 → 调度B
```

#### 模式C：并行→汇聚模式（Parallel→Converge）

```
用户意图 → ┌ [架构师A] → 视角A ┐
           ├ [架构师B] → 视角B ├→ [5号汇聚] → 统一输出
           └ [架构师C] → 视角C ┘
```

适用场景：需要多维度分析同一问题。

规则：
- A/B/C并行工作，互不感知
- 各自产出独立的分析视角
- 5号负责汇聚，解决冲突（见冲突消解协议）
- 汇聚时保留各视角的独特洞察，不强制统一

汇聚协议：
```
5号收集所有视角 → 识别重叠与冲突 → 应用冲突消解协议 → 生成统一建议
```

#### 模式D：分层组合模式（Layered Composition）

```
用户意图 → [5号规划] → ┌ [4号] 输入层 ┐
                       ├ [1号] 分析层 ├→ [5号整合] → 完整平台
                       ├ [2号] 求解层 ├
                       └ [3号] 展示层 ┘
```

适用场景：构建完整平台或端到端工作流。

规则：
- 5号先做整体规划，定义各层的边界和接口
- 各架构师在5号定义的边界内独立工作
- 层间通过5号定义的代数规范接口通信
- 5号负责层间一致性验证

## 四、交接协议（Handoff Protocol）

### 4.1 管线阶段交接

当GCS管线从一个阶段推进到下一个阶段时，可能需要在不同架构师之间交接：

```
IO阶段(4号负责)
  │ 产出: Manager(初始)
  ▼ 交接条件: Manager通过Schema验证
DCM阶段(1号负责)
  │ 产出: DecompositionResult
  ▼ 交接条件: 至少1个连通分量
LGS阶段(1号+2号共同负责)
  │ 产出: StatusReport
  ▼ 交接条件: 约束传播通过(无早期不一致)
Prop阶段(2号负责)
  │ 产出: PropagationResult
  ▼ 交接条件: 弧一致性达成
CDS阶段(2号+3号共同负责)
  │ 产出: SolverReport + Manager(solved)
  ▼ 交接条件: 所有子问题收敛或明确失败
App阶段(5号负责)
  │ 产出: SystemResult
  ▼ 交接条件: 管线完成
```

### 4.2 交接数据包格式

每次交接时，5号负责封装标准化的交接数据包：

```
HandoffPacket {
    from: ArchitectId,           // 来源架构师
    to: ArchitectId,             // 目标架构师
    pipelineStage: Stage,        // 管线阶段
    data: SharedDataContract,    // 共享数据
    context: {                   // 附加上下文
        decisions: Decision[],   // 前序架构师的设计决策
        constraints: String[],   // 前序架构师的约束
        rationale: String[]      // 前序架构师的理由
    },
    validation: {                // 交接验证
        schemaValid: bool,       // 数据契约合规
        invariantsHold: bool,    // 不变量保持
        preconditionsMet: bool   // 前置条件满足
    }
}
```

### 4.3 交接验证规则

| 交接 | 验证规则 | 失败处理 |
|------|----------|----------|
| 4号→1号 | Manager通过Schema验证 | 4号修复后重新交接 |
| 1号→2号 | DecompositionResult非空 | 1号检查图连通性 |
| 2号→2号(Prop→CDS) | 传播无冲突 | 2号报告不一致约束 |
| 2号→3号 | SolverReport可用 | 2号提供失败诊断 |
| 3号→5号 | Manager(solved)参数有效 | 3号标记未求解几何体 |

## 五、冲突消解协议（Conflict Resolution）

### 5.1 重叠区域的冲突类型

#### 冲突1：LGS阶段的DOF分析（1号 vs 2号 vs 5号）

| 架构师 | DOF视角 | 可能冲突 |
|--------|---------|----------|
| 1号 | 图论视角：刚性矩阵秩 → DOF = d|V| - rank(R) | 关注图拓扑结构 |
| 2号 | 数值视角：Jacobian条件数 → 数值稳定性 | 关注求解可行性 |
| 5号 | 代数视角：代数规范 → DOF = Σdof(g) - Σdof_removed(c) - Σrs_adj | 关注接口一致性 |

**消解规则**：三个视角是**互补**而非冲突——
- 1号的图论DOF是**理论值**（通用位置下）
- 2号的数值DOF是**实际值**（当前参数下）
- 5号的代数DOF是**规范值**（接口契约）

**消解策略**：并行展示三个视角，标注差异：

```
┌─────────────────────────────────────────────────────────┐
│ DOF Analysis — Multi-Perspective                         │
├─────────────────────────────────────────────────────────┤
│                                                         │
│ Graph Theory (1号):  DOF = 3×5 - rank(R) = 15-13 = 2  │
│   → UnderConstrained (2 flex modes)                     │
│   → Rigidity matrix rank deficiency: 2                  │
│                                                         │
│ Numerical Analysis (2号): DOF = 2 (confirmed)           │
│   → κ(J) = 4.7 (stable)                                │
│   → Null space dim = 2 (2 infinitesimal flex)           │
│   → Convergence feasible with 2 free parameters         │
│                                                         │
│ Algebraic Spec (5号): DOF = 15 - 7 - 6 = 2             │
│   → geomDOF=15, constrRemoved=7, rsAdj=-6              │
│   → Consistent with graph theory and numerical          │
│                                                         │
│ Consensus: DOF = 2, UnderConstrained                    │
│ Action: Need 2 more constraints for well-constrained    │
└─────────────────────────────────────────────────────────┘
```

#### 冲突2：CDS阶段的求解展示（2号 vs 3号）

| 架构师 | CDS视角 | 可能冲突 |
|--------|---------|----------|
| 2号 | 收敛分析：残差曲线+条件数+收敛速率 | 关注"求解器在做什么" |
| 3号 | 求解动画：几何体移动+约束满足过程 | 关注"几何体在怎么动" |

**消解规则**：2号和3号是**同一过程的不同表示**——
- 2号是**代数表示**（残差空间中的轨迹）
- 3号是**几何表示**（参数空间中的轨迹）

**消解策略**：双视图联动：

```
┌──────────────────────┐  ┌──────────────────────┐
│ 2号: Convergence     │  │ 3号: Geometry Motion  │
│                      │  │                      │
│ Residual             │  │ Monge Top View       │
│  0 ┤*               │  │   ●     ●            │
│    │ *               │  │    \   /             │
│ -2 ┤  **             │  │     ●─●              │
│    │   ***           │  │                      │
│ -6 ┤      *******    │  │ ← 同步动画帧 →       │
│    └──┬──┬──┬──      │  │                      │
│       5  10 15       │  │                      │
│                      │  │                      │
│ [▶ Play] [⏸ Pause]  │  │ [▶ Play] [⏸ Pause]  │
└──────────────────────┘  └──────────────────────┘
         ↑ 播放进度同步 ↑
```

#### 冲突3：图结构修复（1号 vs 4号）

| 架构师 | 修复视角 | 可能冲突 |
|--------|---------|----------|
| 1号 | 刚性视角：添加边使图刚性 | 可能添加过多约束 |
| 4号 | CSP视角：最小修改集 | 可能修改不够刚性 |

**消解规则**：4号是**执行者**，1号是**验证者**——
- 4号提出最小修改方案
- 1号验证修改后的图是否满足刚性条件
- 如果不满足，1号反馈缺失的刚性条件
- 4号在CSP中添加刚性约束，重新求解

**消解策略**：迭代验证循环：

```
4号提出修复方案 → 1号验证刚性 → 
  ├── 满足 → 接受方案
  └── 不满足 → 1号反馈缺失条件 → 4号扩展CSP → 重新求解
```

### 5.2 通用冲突消解原则

| 原则 | 说明 |
|------|------|
| **互补优于统一** | 不同视角是互补的，不强制统一为单一视角 |
| **并行展示** | 冲突时并行展示所有视角，让用户理解差异 |
| **理论优先级** | 当必须选择时，更严格的理论优先（代数规范 > 图论 > 数值经验） |
| **5号仲裁** | 当架构师间无法自行消解时，5号基于全局视野仲裁 |
| **标注差异** | 差异本身是有价值的信息，必须标注而非隐藏 |

## 六、调度流程示例

### 示例1：用户说"帮我生成一个7节点8约束的约束图"

```
Step 1: 5号解析意图 → 识别为图工程任务 → 调度4号
Step 2: 4号执行Henneberg构造 → 生成GCS图 → 输出Manager
Step 3: 5号验证交接 → Manager通过Schema → 调度1号验证
Step 4: 1号分析图结构 → 刚性分析+谱分析 → 输出验证报告
Step 5: 5号汇聚结果 → 
  ├── 验证通过 → 输出最终图
  └── 验证失败 → 调度4号修复 → 回到Step 2
```

### 示例2：用户说"求解这个约束图，我想看全过程"

```
Step 1: 5号解析意图 → 识别为端到端求解任务 → 规划管线
Step 2: 5号调度4号 → 加载图文件 → 产出Manager
Step 3: 5号调度1号 → 可视化图结构+刚性分析 → 产出DecompositionResult
Step 4: 5号调度2号 → 启动管线仪表板 → 监控LGS+Prop+CDS
Step 5: 5号调度3号 → 启动画法几何视图 → 展示求解动画
Step 6: 5号汇聚 → 2号的收敛数据驱动3号的求解动画
Step 7: 求解完成 → 5号汇总所有架构师输出 → 生成完整报告
```

### 示例3：用户说"求解失败了，帮我诊断"

```
Step 1: 5号解析意图 → 识别为诊断任务 → 并行调度1号+2号+3号
Step 2: 1号分析图结构 → 检查刚性条件 → 识别结构缺陷
Step 3: 2号分析数值行为 → 检查条件数+收敛模式 → 识别数值问题
Step 4: 3号分析几何位置 → 检查退化/冲突 → 识别几何问题
Step 5: 5号汇聚三个视角 → 
  ├── 1号: 图不满足Hendrickson条件(缺2条边)
  ├── 2号: κ(J)=10^7, Jacobian近奇异
  └── 3号: Point#0和Point#1初始位置重合(退化)
Step 6: 5号消解冲突 → 三者一致指向同一根源
Step 7: 5号调度4号 → 基于诊断结果修复图
```

### 示例4：用户说"我要构建GCS的完整Web平台"

```
Step 1: 5号解析意图 → 识别为平台构建任务 → 分层组合模式
Step 2: 5号规划层次 →
  ├── 输入层: 4号负责(图工程+文件加载)
  ├── 分析层: 1号负责(图可视化+刚性分析)
  ├── 求解层: 2号负责(管线仪表板+收敛分析)
  ├── 展示层: 3号负责(画法几何视图+求解动画)
  └── 整合层: 5号负责(跨层桥接+状态管理)
Step 3: 5号定义代数规范接口 → 各层边界
Step 4: 各架构师在边界内独立设计
Step 5: 5号验证层间一致性 → 整合为完整平台
```

## 七、调度规则总结

### 7.1 调度决策树

```
用户意图
├── 涉及图结构/拓扑/刚性？
│   └── 是 → 1号(主) + [4号(如需修复)]
├── 涉及求解过程/收敛/数值？
│   └── 是 → 2号(主) + [1号(图分析) + 3号(几何位置)]
├── 涉及几何位置/空间/投影？
│   └── 是 → 3号(主) + [2号(求解状态)]
├── 涉及图生成/验证/修复？
│   └── 是 → 4号(主) + [1号(验证)]
├── 涉及整体平台/跨层/端到端？
│   └── 是 → 5号(主) + [1+2+3+4号(按需)]
└── 不确定？
    └── 5号(默认) → 5号分析后调度
```

### 7.2 5号的调度职责清单

| 职责 | 说明 |
|------|------|
| 意图解析 | 将用户意图映射到架构师调度方案 |
| 交接管理 | 管控架构师之间的数据交接 |
| 冲突消解 | 仲裁架构师间的视角冲突 |
| 一致性验证 | 验证跨架构师的输出一致性 |
| 全局规划 | 为复杂任务规划多架构师协作方案 |
| 默认兜底 | 不确定时由5号先行分析 |

### 7.3 1-4号的执行职责清单

| 职责 | 说明 |
|------|------|
| 领域专精 | 在自己的理论基础上提供最深度的分析 |
| 数据契约遵守 | 产出必须符合共享数据契约 |
| 决策理由 | 每个设计决策附理由注释 |
| Anti-Pattern遵守 | 不违反自己定义的Anti-Pattern |
| 不越界 | 不直接调用其他架构师，通过5号协调 |

### 7.4 禁止事项

| 禁止 | 理由 |
|------|------|
| 1-4号之间直接调用 | 避免环形依赖和不可控的级联效应 |
| 跳过5号直接组合 | 5号负责一致性验证，跳过可能导致不一致 |
| 架构师越界决策 | 每个架构师只在自己的专业领域内决策 |
| 忽略冲突 | 冲突必须标注和消解，不能隐藏 |
| 强制统一视角 | 不同视角的差异是有价值的信息 |

## 八、与GCS管线的完整调度映射

```
用户操作               调度链
─────────────────────────────────────────────────────
加载图文件         →   4号 → [1号验证]
生成新图           →   4号 → [1号验证] → [2号DOF预检]
查看图结构         →   1号
分析图刚性         →   1号 → [4号修复(如需)]
启动求解           →   5号 → 2号(仪表板) + 3号(几何视图)
监控求解过程       →   2号
查看求解动画       →   3号 ← 2号提供收敛数据
求解失败诊断       →   5号 → 1号+2号+3号(并行) → 汇聚
修复图后重求解     →   4号(修复) → 1号(验证) → 2号+3号(重求解)
对比两个场景       →   5号 → 1号+2号+3号(各维度对比)
构建完整平台       →   5号(规划) → 4号+1号+2号+3号(分层)
```
